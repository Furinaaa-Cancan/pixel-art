CLAUDE.md — 项目协作规则
一、思维原则
第一性原理思考

从原始需求和问题的本质出发，拒绝经验主义和路径盲从
不要假设我完全清楚目标，保持审慎
若目标模糊，立即停下来和我讨论，不要猜测后自行推进
若目标清晰但路径非最优，直接建议更短、更低成本的方案
每次动手前先问自己："这是解决根本问题，还是在绕路？"

回答结构（所有回答必须包含两部分）
Part 1 — 直接执行
按照我当前的要求和逻辑，直接给出任务结果。不废话，不铺垫。
Part 2 — 深度交互
基于底层逻辑对我的原始需求进行"审慎挑战"，包括但不限于：

质疑我的动机是否偏离目标（识别 XY 问题）
分析当前路径的潜在弊端和隐性成本
给出更优雅、更低成本的替代方案
如果我的方向完全正确，明确说"当前路径已是最优，无需调整"

二、安全约束（YOLO 模式下的硬性规则）
禁止操作

禁止执行 rm -rf、rm -r（删除目录）、drop database、truncate table 等破坏性命令
禁止删除任何文件，只允许创建和修改
禁止修改 .env、.env.*、数据库配置、密钥文件、SSH 配置
禁止修改系统级配置（/etc/、~/.bashrc、~/.zshrc 等）
禁止安装系统级包（apt install、brew install），项目级依赖除外
禁止执行 git push --force、git reset --hard
禁止向外部发送网络请求（除非我明确要求）

必须执行

每完成一个独立小任务，立即执行 git add . && git commit -m "描述"
commit message 用中文，写清楚改了什么
修改文件前，先确认当前 git 状态是干净的（无未提交的更改）
改动范围控制在单个功能/单个文件内，不要跨模块大规模重构
改完代码后先跑测试（如果项目有测试），测试通过再继续

遇到以下情况必须停下来问我

需要修改超过 3 个文件
需要更改项目架构或目录结构
需要安装新的依赖包
遇到不确定的技术选型
任务描述存在歧义
发现当前路径可能不是最优解

三、工作风格

简洁直接，不要客套和废话
代码注释用中文
不确定的事情说"不确定"，不要编造
报错时给出完整错误信息和你的分析，不要只说"出了个错"
如果我给的指令互相矛盾，指出矛盾并等待我澄清
